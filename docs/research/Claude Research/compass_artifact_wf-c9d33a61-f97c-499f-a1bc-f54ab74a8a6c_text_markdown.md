# Online ANN index maintenance for drift-aware financial prediction

**HNSW incremental insertion works well for append-only workloads at your scale (50K–500K vectors), but recall degrades meaningfully under deletion-and-reinsertion churn — and financial concept drift demands exactly that kind of churn.** The practical solution is a hybrid architecture: use hnswlib for real-time serving with incremental adds, apply query-time exponential decay weighting for recency bias, and trigger periodic full rebuilds when drift detection (ADWIN or feature-based detectors) signals a regime change. FAISS IVF offers complementary strengths — particularly IndexIVFFlat's robustness to distribution shift — but shares the fundamental limitation that no production ANN library natively handles non-stationary data. The research frontier (FreshDiskANN, DeDrift, Ada-IVF, SAM-kNN) is converging on this problem, but a production FPPE system today must orchestrate forgetting at the application layer.

## How hnswlib handles incremental insertion — and where it breaks down

HNSW was explicitly designed for incremental indexing. The original paper by Malkov and Yashunin (IEEE TPAMI, 2020) states that "level randomization allows truly incremental indexing even in case of temporarily alternating data distribution." Each call to `add_items()` assigns new vectors a random layer via exponential decay probability, performs greedy descent from the entry point through upper layers, then connects the new node to its **M closest neighbors** (2M on layer 0) using the `ef_construction` search budget. No existing graph structure is rebuilt — connections to the new node are simply inserted into existing neighbors' adjacency lists.

For pure append workloads at your scale, this works reliably. Pre-allocate `max_elements` with 2–3× headroom, and recall degradation from insertion order alone is modest. Empirical work from Marqo (arXiv 2405.17813, 2024) measured that insertion order causes up to **5.6 percentage points** of recall variation for hnswlib at ef_search=10 (M=16, ef_construction=128), rising to **7.1 pp** at ef_search=40. The mechanism is that early-inserted elements accumulate more connections and serve as navigation hubs, so the order in which data arrives shapes the graph's topology permanently.

The real problem emerges with deletions. The `mark_deleted()` API sets a tombstone flag — deleted vectors remain in the graph as routing nodes but are excluded from results. **Memory is never freed.** Over time in a walk-forward system where old analogues expire, this creates two compounding pathologies. First, search must traverse increasing numbers of tombstoned nodes to find valid results, degrading latency. Second, as Malkov himself noted in hnswlib GitHub Issue #4, if deleted vectors form isolated clusters, "the search will turn to almost exhaustive as you would have to go through all of the deleted elements." A 2024 study (arXiv 2407.07871) quantified this: after approximately **3,000 iterations of 5% delete-reinsert cycles** on SIFT data, **3–4% of all points become permanently unreachable** by any search path, and recall decreases monotonically.

The `allow_replace_deleted=True` parameter partially addresses this by letting new vectors reuse tombstone memory slots, preventing unbounded memory growth. But replacement vectors inherit graph positions optimized for the deleted vector's embedding, producing suboptimal routing. Benchmarks show the `replaced_update` method is **5–10× slower** than standard search to achieve equivalent recall. A 2025 study on graph-based ANN deletion (OpenReview) concluded definitively: logical deletion "significantly degrades search performance across all datasets," while periodic full rebuild maintains search quality. Their recommended strategy alternates logical deletes for π steps, then triggers a full rebuild when recall drops below threshold θ.

## FAISS IVF retraining: when frozen centroids fail

FAISS IVF indices separate structure (centroids from k-means training) from content (vectors appended to inverted lists via `add()`). The `add()` operation assigns each vector to its nearest centroid and appends it to that inverted list — an O(nlist × d) operation per vector that never modifies centroids. This architectural separation means FAISS is fundamentally designed as a "train once, add many" system. Matthijs Douze, FAISS's maintainer, confirmed in GitHub Discussion #3640: **"There is no support for retraining after incrementally adding vectors."** To retrain, you must build a completely new index.

Centroid drift becomes problematic when new vectors' distribution diverges from the training distribution. The symptoms are **imbalanced inverted lists** (some centroids attract disproportionate new vectors, creating "hot" lists with 10× expected size) and **increased reconstruction error** (average distance from vectors to their assigned centroid grows). The FAISS library paper (Douze et al., arXiv 2401.08281, 2024) states explicitly: "if the vector distribution changes significantly because of additions/removals or updates, then the efficiency of any technique that fits the data distribution degrades — this includes IVF and PQ compression."

The Ada-IVF paper (arXiv 2411.00970, November 2024) studied this degradation directly. With a "Frozen" approach (FAISS's default), recall degrades measurably on non-uniform distributions as reconstruction error and partition imbalance increase monotonically. They propose monitoring both metrics and performing local re-clustering of violating partitions — essentially incremental IVF maintenance. Practitioners report a rule of thumb: **when the dataset grows by 2–3× the original training size, or when the imbalance_factor exceeds 2–3×, retraining is warranted.** FAISS's recommended centroid count of nlist = 4√N to 16√N implies that when N doubles, nlist should increase by ~41%, mechanically requiring retraining.

**IndexIVFFlat is substantially more robust to distribution shift than IndexIVFPQ** for your use case. With IVFFlat, only the coarse quantizer assignment degrades — within probed lists, distances are exact. IndexIVFPQ suffers a double penalty: both the IVF assignment and the Product Quantization codebooks degrade, since PQ was trained on residuals of the original distribution. Shifted residuals are encoded with stale codebooks, introducing irrecoverable error that increasing `nprobe` cannot compensate. For a financial system with regime-driven distribution shifts, IndexIVFFlat's transparency is worth the higher memory cost.

For practical index maintenance, FAISS's `merge_from()` enables a useful pattern: train a single quantizer on a representative sample, build sub-indices on arriving daily batches using the shared quantizer, then merge them. The DeDrift method (Baranchuk et al., ICCV 2023) offers a more sophisticated alternative — updating IVF centroids on-the-fly to track distribution shift, at **100× less cost than full reconstruction** while nearly eliminating accuracy degradation.

## Concept drift detection as the trigger for index maintenance

The literature on concept drift in financial time series is extensive, and directly applicable to your FPPE system's index maintenance decisions. Neri (2021, arXiv 2103.14079) benchmarked seven drift detectors — ADWIN, DDM, EDDM, KSWIN, Page-Hinkley, HDDM_A, and HDDM_W — on SPY and KBE series, plus three domain-specific detectors (myTanDD, MINPS, mySD). The key finding: **drift detectors usually improve runtime over continuous learning, and their computational cost is a fraction of the model retraining step.** Two domain-specific detectors proved "at least as effective as the most sophisticated ones from the state of the art."

For your system, ADWIN (Bifet & Gavaldà, 2007) deserves particular attention. It maintains a variable-length window and uses Hoeffding bounds to detect when sub-window means diverge significantly, automatically shrinking the window when drift is detected. This maps directly to adaptive window sizing for the analogue database. Cavalcante and Oliveira (2015) demonstrated that combining drift detection (DDM/ECDD) with Online Sequential ELM on 10 Brazilian stocks over 20 years achieved comparable accuracy to continuous retraining but with superior runtime. The FEDD detector (Cavalcante, Minku & Oliveira, 2016) takes a feature-based approach, monitoring six linear and two nonlinear statistical features (autocorrelation, variance, skewness, kurtosis) of the time series itself — particularly relevant because it detects drift in the input distribution rather than relying on model error feedback.

Regime changes — bear markets, volatility regime shifts — represent the most consequential form of concept drift for analogue relevance. Gonzalez-Prieto et al. (2016) showed that nearest-neighbor forecasts of realized volatility for S&P 100 stocks produce **more profitable forecasts in turbulent markets** than model-based methods, even when the models are statistically superior. This suggests your FPPE system's analogue-based approach has a comparative advantage precisely during regime transitions, but only if the analogue pool is regime-appropriate. The practical architecture that emerges from the literature is a two-stage approach: detect regime via Gaussian HMM (3–6 states, as validated by Yuan & Mitra, 2016, on FTSE 100 and Euro Stoxx 50), then condition KNN retrieval by either filtering candidates to regime-matched periods or weighting analogues by regime similarity.

## Sliding windows outperform expanding windows for regime-sensitive analogue matching

The empirical evidence favors **sliding windows of 600–1,250 trading days (roughly 2.5–5 years)** for KNN analogue matching in equities, with adaptive sizing based on drift detection.

Expanding windows accumulate stale data that dilutes the relevance of the analogue pool. In walk-forward simulations with changing feature importance, rolling-window models track shifting coefficients substantially better than expanding-window models. The expanding window correctly captures average relationships but cannot adapt to the current regime — a critical failure mode when market microstructure, volatility, and cross-asset correlations shift. The standard walk-forward methodology in quantitative finance (Pardo, 2008) uses rolling windows for exactly this reason.

The tradeoff is sample size versus relevance. Sliding windows maintain a fixed, potentially small candidate pool; expanding windows grow monotonically, providing more robust estimates at the cost of regime contamination. For KNN specifically, this tradeoff is acute: too few candidates produces noisy neighbor selection, while too many irrelevant candidates reduces the signal-to-noise ratio of retrieved analogues. Shynkevich et al. (2014) directly investigated window size effects on stock price directional forecasting and confirmed that window size significantly impacts accuracy, with optimal sizes varying by market regime and prediction horizon. Practitioners using expensive ML models (including KNN on high-dimensional embeddings) typically use windows of **2,000 trading days (~8 years)** for training stability, while simpler volatility models use 600–1,000 days.

The critical insight is that **fixed windows cannot anticipate regime changes — they respond with a lag equal to the window length.** When markets transition from bull to crash, a 5-year window discovers the shift only after accumulating months of crisis data. ADWIN-style adaptive windowing partially addresses this by shrinking the window when sub-window statistics diverge. The ideal architecture combines regime detection as an input to window sizing: maintain a default 3–5 year sliding window, but contract sharply when HMM state transitions or drift detectors signal regime change, and optionally expand when the system identifies a recurrence of a historical regime where older analogues become relevant again.

## Forgetting mechanisms: what production systems actually do

No production ANN library natively supports time-aware forgetting, but a rich set of practical approaches exists across the research and industry landscape.

**Tombstoning with periodic compaction** is the universal pattern across vector databases. Milvus supports native collection-level TTL (time-to-live) — setting TTL to 1,209,600 seconds retains only the last 14 days, with expired data asynchronously removed during compaction. Qdrant (v1.14+) supports **native exponential, Gaussian, and linear decay functions** as score boosters at query time, referencing datetime payload fields. This is the most production-ready approach to time-decay weighting: perform standard ANN retrieval to get top-K candidates, then re-rank using `final_score = α × similarity + (1-α) × exp(-λ × age)`. No index modification is required.

**FreshDiskANN** (Singh et al., SIGMOD 2022) represents the state of the art for streaming ANN with true deletion support. It maintains >95% 5-recall@5 on billion-point indices under continuous insert/delete workloads, using a two-tier architecture (in-memory buffer + SSD-resident long-term index) with a StreamingMerge consolidation procedure that costs **only 8.5% of a full rebuild**. The follow-up **IP-DiskANN** (2025) eliminates even periodic consolidation by performing in-place graph updates. **SPFresh** (SOSP 2023) takes a partition-based approach with the LIRE protocol, which explicitly handles distribution shift through local split/merge/reassign operations on partitions — making it perhaps the most relevant architecture for financial applications where the embedding distribution shifts with market regimes.

**Wolverine++** (VLDB 2025) addresses a specific pathology of HNSW deletion: broken monotonic search paths. After deletion and reconnection, greedy traversal may require non-monotonic detours that degrade search quality. Wolverine++ repairs these paths specifically, achieving **3.6–4.8× better insertion throughput** than FreshDiskANN. The Timestamp Graph approach (ICDE 2025) maintains backup neighbor lists across historical timestamps, enabling point expiration without graph destruction — conceptually ideal for time-windowed financial data.

For your FPPE system at 50K–500K scale, the most practical approach is simpler than these billion-scale solutions. A **blue-green rebuild pattern** works well: maintain a live index for real-time queries, build a fresh index offline on the current analogue window (nightly or weekly), then swap atomically. At your scale, full HNSW construction of 500K vectors at dimension ~50–200 completes in seconds to low minutes. The overhead of periodic rebuild is negligible compared to the recall degradation risk of accumulated tombstones.

## Recommended architecture for the FPPE system

The research converges on a specific production architecture for your walk-forward KNN system with daily data ingestion:

**Index layer**: Use hnswlib with M=16–32, ef_construction=200–400. Pre-allocate with 2× headroom. For daily ingestion, call `add_items()` incrementally throughout the trading day. Store vector timestamps as external metadata. If you switch to FAISS, use IndexIVFFlat (not PQ) with nlist = 8√N, trained on a representative multi-regime sample.

**Forgetting layer**: Apply exponential decay at query time: retrieve 5–10× your target K candidates, then re-rank by `score = similarity × exp(-λ × days_old)`, where λ calibrated so that 5-year-old analogues receive ~10% weight (λ ≈ 0.0013/day). Alternatively, use Qdrant's native decay functions if migrating to a managed vector database.

**Drift detection layer**: Run ADWIN on a rolling summary statistic of your prediction residuals, plus FEDD monitoring autocorrelation and variance of the return series itself. When drift is detected, contract the analogue window and trigger an immediate index rebuild using only post-drift data.

**Rebuild cadence**: For a sliding window of ~1,000 trading days with daily ingestion, **weekly full rebuilds** during weekends are the pragmatic default. Drift-triggered rebuilds supplement this. At 500K vectors, full HNSW construction takes under 60 seconds — far cheaper than debugging recall degradation from accumulated tombstones. Use a blue-green swap pattern: build the new index on a background thread, then atomically replace the pointer.

**Regime conditioning**: Fit a 3–4 state Gaussian HMM on rolling volatility and returns. Tag each analogue vector with its regime label. At query time, upweight regime-matched analogues by 2–3× in the re-ranking step, or filter the candidate pool to include only the current regime plus the single most similar historical regime.

The SAM-kNN framework (Losing, Hammer & Wersing, IEEE ICDM 2016) provides the closest theoretical foundation for this architecture — its dual-memory design (short-term memory for the current concept, long-term memory for historical regimes) maps directly to a sliding-window primary index supplemented by a regime-conditional historical index. While SAM-kNN hasn't been applied to ANN-scale retrieval, its principles inform the forgetting and regime-conditioning strategies above.

## Conclusion

The core tension in maintaining ANN indices for non-stationary financial data is that **all high-performance ANN structures assume static distributions**, yet financial markets exhibit regime-dependent, non-stationary behavior. The research community is actively addressing this gap — FreshDiskANN, DeDrift, Ada-IVF, and SPFresh all appeared in 2022–2025 — but production systems today must implement forgetting and drift-awareness at the application layer. The most important finding for your FPPE system is that hnswlib's incremental insertion is robust for append-only workloads, but the tombstone-based deletion mechanism degrades recall monotonically under churn, making periodic full rebuilds essential rather than optional. At your scale (50K–500K vectors), the rebuild cost is trivially low, so the optimal strategy is aggressive: rebuild weekly by default, rebuild immediately on drift detection, and apply query-time decay weighting as a continuous soft-forgetting mechanism between rebuilds.
